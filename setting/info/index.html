配置文件说明#

配置文件mooder/settings.py中包含的配置项的说明。

说明一下，如果你是使用docker-compose一键安装的Mooder，可以无需修改任何选项，默认会生成一个安全的生产环境。如果你还是需要手工修改配置，请执行 docker-compose exec web /bin/bash 进入容器内进行修改。

1. 如何自定义配置#

Mooder包含一个基础配置文件（mooder/settings.py）和一个生产环境配置文件(mooder/settings_production.py)。二者是继承关系，settings_production.py继承所有settings.py定义的配置，并通过定义相同名字的变量来修改配置。

如，默认settings.py中定义DEBUG=True，即开启调试模式；而settings_production.py中定义DEBUG=False，就覆盖了settings.py的设置，关闭了Django的调试模式。

所以，建议你不要修改mooder/settings.py，直接修改并使用mooder/settings_production.py。

如果你需要自定义配置（不是用默认的生产环境配置文件），你可以新建一个mooder/settings_custom.py，并导入所有settings.py的配置项：

from mooder.settings import *

# 再在后面修改你自定义的配置，如DEBUG=True
DEBUG = True

Mooder启动的时候，会默认加载mooder/settings.py作为配置文件。为了使用自定义配置文件，我们需要设置一个环境变量，指定自定义配置文件的位置：

# DJANGO_SETTINGS_MODULE 环境变量的值是Python包格式
export DJANGO_SETTINGS_MODULE=mooder.settings_custom

# 运行Mooder
python3 manage.py runserver

Windows下使用set来指定环境变量：

set DJANGO_SETTINGS_MODULE=mooder.settings_custom
py -3 manage.py runserver

如果你使用系统自带的生产环境配置文件（mooder/settings_production.py），也需要定义该环境变量。

在指定了环境变量DJANGO_SETTINGS_MODULE后，Mooder就会使用DJANGO_SETTINGS_MODULE指向的配置文件。

2. 配置项说明#

大部分配置项在Django的 官方文档 中都有说明，我这里不会说的很详细，只会说一些比较关键的项目。

DEBUG#

是否是调试模式。注意，生产环境下一定要设置为False，否则可能造成安全问题！

SECRET_KEY#

加密与签名的密钥，一个随机字符串。

查看settings.py文件中的定义：

if os.path.exists(os.path.join(BASE_DIR, '.secretkey')):
    with open(os.path.join(BASE_DIR, '.secretkey'), 'rb') as f:
        SECRET_KEY = f.read()

可见，如果.secretkey文件存在，则SECRET_KEY从.secretkey中读取。否则默认是没有配置SECRET_KEY的，你必须手工定义它。也就是说，如果当前目录下存在.secretkey文件，则无需手工定义SECRET_KEY。

你可以用openssl简单地生成一个随机字符串保存在.secretkey文件中，无需再用脸滚键盘：

sudo apt-get install openssl
openssl rand -out .secretkey 64

ALLOWED_HOSTS#

允许访问的host头，可为多个，如： ['example.com', 'www.example.com']。不限制则设置为['*']。

调试模式关闭时，必须设置这个选项。

DATABASES#

数据库配置选项，详见 Django文档 和前一章说明。

EMAIL_HOST_USER / EMAIL_BACKEND / MAILGUN_ACCESS_KEY / MAILGUN_SERVER_NAME#

邮件客户端配置，详见 Django文档

Mooder默认使用Mailgun作为邮件发送服务商，在你的Mailgun后台可以看到相应域名的API BASE URL和KEY，将其填入如下配置中即可

EMAIL_HOST_USER = '发送邮件的邮箱'
EMAIL_BACKEND = 'django_mailgun.MailgunBackend'
MAILGUN_ACCESS_KEY = 'Mailgun KEY'
MAILGUN_SERVER_NAME = 'Mailgun API Base URL'

当然你也可以通过设置EMAIL_BACKEND来使用其他邮件发送方式，如SMTP：

EMAIL_BACKEND = 'django.core.mail.backends.smtp.EmailBackend'
EMAIL_USE_TLS = True
EMAIL_HOST = 'smtp.gmail.com'
EMAIL_PORT = 587
EMAIL_HOST_USER = 'example@gmail.com'
EMAIL_HOST_PASSWORD = 'password'

STATIC_URL / STATIC_ROOT#

静态文件相关配置。

STATIC_URL是在URL中匹配静态文件请求的前缀，STATIC_ROOT是静态文件实际保存的目录。

例如，Nginx配置如下：

...
location ^~ /static/
{
    root /var/www/html;
    expires 30d;
}
location / {
    proxy_set_header X-Real-IP  $remote_addr;
    proxy_pass http://127.0.0.1:8000;
}
...

当用户请求http://example.com/static/css/app.css的时候，Nginx会直接读取文件/var/www/html/static/css/app.css并发送给用户（不会经过Mooder处理）。

当用户请求的PATH前缀不是/static/的时候，将会转发给本地的8000端口，交给Mooder处理。

所以，mooder.settings中STATIC_URL就是该URL前缀，而STATIC_ROOT就是实际储存静态文件的本地路径。当配置好这个选项后，我们需要运行 python3 manage.py collectstatic 将所有静态文件生成到STATIC_ROOT目录中。从此刻起，静态文件就与Mooder无关了，你需要使用Nginx或Apache等中间件来显示静态文件。

另外，在DEBUG模式下，我们无需生成静态文件，由Mooder显示静态文件。

MEDIA_URL / MEDIA_ROOT#

与STATIC_URL类似，MEDIA_ROOT是储存附件文件的目录，而MEDIA_URL则是这个URL前缀。

附件文件初始化的时候不存在，是后期网站运行过程中用户上传的附件、头像等文件，所以无需运行collectmedia来生成。

例如，配置如下：

STATIC_URL = '/static/'
STATIC_ROOT = '/var/www/html/static'

MEDIA_URL = '/media/'
MEDIA_ROOT = '/data/media'

相对应的Nginx配置文件则为：

...
location ^~ /static/
{
    root /var/www/html;
    expires 30d;
}
location ^~ /media/
{
    root /data;
}
location / {
    proxy_set_header X-Real-IP  $remote_addr;
    proxy_pass http://127.0.0.1:8000;
}
...

如果你是使用docker-compose一键安装，则无需配置这些选项。自动化脚本会创建硬盘卷映射到/data目录下，即使容器此后更新或删除，都不会影响附件文件，你可以很容易地对其进行迁移；同时，自动化脚本也会自动映射^/media/.*请求到该目录中，无需再单独配置。

TIME_ZONE#

时区，默认为'Asia/Shanghai'

LANGUAGE_CODE#

语言，暂时仅支持中文，请勿修改。（修改以后将会成为中文和其他语言混杂的奇怪形式）

USER_LEVEL_RANGE#

用户等级名称与范围。默认如下：

USER_LEVEL_RANGE = (
    ((0, 30), ('anonymous', '路人甲'), 'default'),
    ((30, 100), ('practice', '见习白帽子'), 'secondary'),
    ((100, 300), ('general', '正式白帽子'), 'primary'),
    ((300, 1000), ('senior', '高级白帽子'), 'success'),
    ((1000, 2000), ('supreme', '核心白帽子'), 'warning'),
    ((2000, -1), ('god', '妇科圣手'), 'danger')
)

解释一下，(0,30)表示rank在0到30（含0不含30）的时候等级为“路人甲”；“anonymous”是该等级的识别名称，请勿修改；“default”是显示颜色，暂时只有上述6种颜色，也就是说如果你增加了一个等级，将必须有两个等级的颜色相同。

如果你要对该配置进行修改，建议只修改“等级范围”和“名称”，例如修改为如下：

USER_LEVEL_RANGE = (
    ((0, 100), ('anonymous', '小学生'), 'default'),
    ((100, 300), ('practice', '初中生'), 'secondary'),
    ((300, 1000), ('general', '高中生'), 'primary'),
    ((1000, 5000), ('senior', '大学生'), 'success'),
    ((5000, 20000), ('supreme', '研究生'), 'warning'),
    ((20000, -1), ('god', '博士'), 'danger')
)

最后一个等级的范围是(xxx, -1)，指rank大于xxx的用户，-1请不要修改。

SITE#

网站SEO信息，默认为如下：

SITE = {
    'title': '安全团队贡献平台',
    'description': '一个团队使用的贡献平台',
    'keyword': '安全,团队,贡献'
}

title即为网站总标题。

ACCOUNT_ACTIVATION_DAYS#

激活邮件有效期，单位为天。默认为3天：

ACCOUNT_ACTIVATION_DAYS = 3

其他选项，如果需要自定义，可以查看Django文档。
